## Executive Summary

SpectraCore v1.2.0 represents a **platform expansion and feature enrichment** release. Building on the solid foundation of v1.1.0 (bandpass/bandstop filters across all platforms), v1.2.0 aims to:

1. **Expand platform support** to mobile (iOS, Android) and web (WebGL/WASM)
2. **Enhance filter capabilities** with Bessel and Parks-McClellan FIR filters
3. **Add advanced analysis** functions (MFCC, spectral features)
4. **Optimize performance** with SIMD and Unity Burst integration
5. **Improve UX** with preset systems and enhanced visualization

**Strategic Focus:** Enable SpectraCore for mobile and web games while maintaining desktop-class performance.

**Current State:**
- Native library: C++17 (compatible)

- Unity plugin architecture: Supports iOS

**Technical Requirements:**


2. **Platform-Specific Plugins:**
   - Location: `unity/Runtime/Plugins/iOS/libspectra.a` (static library for iOS)
   - Unity meta configuration: iOS native plugin settings

3. **Build Process:**
   ```bash
   # Cross-compile from macOS
   cmake -B build-ios -G Xcode \
       -DCMAKE_TOOLCHAIN_FILE=cmake/ios.toolchain.cmake \
       -DPLATFORM=OS64  # iOS devices
   cmake --build build-ios --config Release
   ```

4. **Challenges:**
   - **Static vs Shared:** iOS prefers static libraries (.a) over dylibs
   - **Bitcode:** May need bitcode support (deprecated iOS 14+, but check)
   - **App Store:** No JIT compilation restrictions (not relevant for native C++)

**Implementation Effort:** 3-5 days (macOS build machine required)

### 2. WebGL/WASM Platform

**Current State:**
- Native C++ library (needs WASM compilation)
- Unity WebGL uses Emscripten toolchain
- No existing WebGL support

**Technical Requirements:**

1. **Emscripten Build:**
   ```bash
   # Install Emscripten SDK
   git clone https://github.com/emscripten-core/emsdk.git
   cd emsdk
   ./emsdk install latest
   ./emsdk activate latest

   # Build with Emscripten
   emcmake cmake -B build-wasm -DCMAKE_BUILD_TYPE=Release
   cmake --build build-wasm
   ```

2. **Output:**
   - `libspectra.wasm` - WebAssembly binary
   - `libspectra.js` - JavaScript glue code
   - Location: `unity/Runtime/Plugins/WebGL/`

3. **Unity Integration:**
   - Use `[DllImport("__Internal")]` for WebGL
   - Conditional compilation: `#if UNITY_WEBGL && !UNITY_EDITOR`

4. **Challenges:**
   - **Threading:** WebGL has limited threading (Web Workers)
   - **Memory:** WASM memory limits (can be mitigated with dynamic growth)
   - **File System:** No direct file I/O (use IndexedDB/virtual FS)
   - **Performance:** ~50-70% of native speed (still real-time capable)

5. **Optimizations:**
   - Enable SIMD: `-msimd128` (WASM SIMD proposal, ~90% browser support)
   - Threading: `-pthread` (if needed, with caveats)
   - Size: `-Oz` for smaller binary (~200KB after compression)

**Code Example:**
```csharp
// Unity C# wrapper - platform-conditional

    [DllImport("__Internal")]
#else
    [DllImport("spectra")]
#endif
private static extern int spectra_butter(/* ... */);
```

**Implementation Effort:** 5-8 days (WASM experience recommended, testing critical)

### 3. Bessel Filters

**Overview:**
Bessel filters optimize for **linear phase** (constant group delay), resulting in minimal signal distortion. Ideal for audio, pulse/transient preservation.

**Mathematical Basis:**
Bessel filter poles derived from **Bessel polynomials** (Thomson filters):
- Maximally flat group delay
- Gentle rolloff (worse than Butterworth)
- Excellent transient response

**Implementation:**

1. **Algorithm:**
   - Compute Bessel polynomial coefficients (lookup table or recurrence)
   - Find poles (roots of Bessel polynomial)
   - Apply lowpass-to-X transformations (same as Butterworth)
   - Bilinear transform to digital domain

2. **Code Structure:**
   ```cpp
   // native/src/filters/filter_design.cpp

   // Bessel polynomial coefficients (pre-computed, order 1-10)
   static const double bessel_poly_coeffs[10][11] = { /* ... */ };

   FilterCoefficients bessel(int order, float cutoff, FilterType type) {
       // 1. Get Bessel poles from polynomial
       std::vector<std::complex<double>> poles = bessel_poles(order);

       // 2. Normalize to cutoff frequency
       for (auto& p : poles) p *= cutoff;

       // 3. Apply lowpass-to-X transform
       if (type == FilterType::Highpass) {
           poles = lowpass_to_highpass(poles, cutoff);
       }
       // ... bandpass/bandstop similar to Butterworth

       // 4. Bilinear transform
       return bilinear_zpk(/* ... */);
   }
   ```

3. **Dual-Frequency Overload:**
   ```cpp
   FilterCoefficients bessel(int order, float low_freq, float high_freq,
                             FilterType type);
   ```

4. **Unity C# API:**
   ```csharp
   // DSP.cs
   public static (float[] b, float[] a) Bessel(int order, float freq,
                                                FilterType type);
   public static (float[] b, float[] a) Bessel(int order, float lowFreq,
                                                float highFreq, FilterType type);
   ```

**Implementation Effort:** 3-4 days (similar to existing filter types)

**MATLAB Validation:**
```matlab
% MATLAB does have Bessel filters
[b, a] = besself(4, 2*pi*1000);  % Analog Bessel
[bz, az] = bilinear(b, a, 44100);  % Digital conversion
```

### 4. Parks-McClellan FIR Filter

**Overview:**
Optimal equiripple FIR filter design using **Remez exchange algorithm**. Provides:
- Minimum filter order for given specs
- Equiripple passband/stopband (optimal in minimax sense)
- Arbitrary frequency response (multiband)

**Use Cases:**
- Sharp cutoff FIR filters
- Custom frequency responses (notch, multi-band)
- Linear phase applications

**Implementation:**

1. **Algorithm Choice:**
   - **Option A:** Implement Remez exchange from scratch (~500 lines)
   - **Option B:** Port existing implementation (SciPy `remez`, MATLAB `firpm`)
   - **Recommendation:** Option B (port from SciPy, well-tested)

2. **SciPy Reference:**
   ```python
   # scipy.signal.remez
   from scipy import signal
   h = signal.remez(numtaps=51, bands=[0, 0.2, 0.3, 0.5],
                    desired=[1, 0], weight=[1, 2])
   ```

3. **API Design:**
   ```cpp
   // native/include/spectra/filters/filter_design.hpp

   struct FrequencyBand {
       float start;  // Normalized frequency (0-1)
       float stop;
       float desired_gain;  // 0 or 1
       float weight;  // Relative importance
   };

   FilterCoefficients firpm(int num_taps,
                            const std::vector<FrequencyBand>& bands);
   ```

4. **Unity C# API:**
   ```csharp
   // DSP.cs
   public struct FrequencyBand {
       public float Start;
       public float Stop;
       public float DesiredGain;
       public float Weight;
   }

   public static float[] Firpm(int numTaps, FrequencyBand[] bands);
   ```

5. **Example Usage:**
   ```csharp
   // Design a bandpass FIR: passband 0.2-0.3, stopbands 0-0.15 and 0.35-0.5
   var bands = new[] {
       new FrequencyBand { Start = 0.0f, Stop = 0.15f, DesiredGain = 0, Weight = 1 },
       new FrequencyBand { Start = 0.2f, Stop = 0.3f, DesiredGain = 1, Weight = 1 },
       new FrequencyBand { Start = 0.35f, Stop = 0.5f, DesiredGain = 0, Weight = 1 }
   };
   float[] h = DSP.Firpm(51, bands);
   ```

**Implementation Effort:** 7-10 days (complex algorithm, extensive testing required)

**Challenges:**
- Remez algorithm convergence (may fail for unrealistic specs)
- Numerical stability (double precision required)
- MATLAB equivalence validation

### 5. SIMD Optimization

**Overview:**
Vectorize critical DSP operations using CPU SIMD instructions for 2-8× speedup.

**Platform SIMD Support:**

| Platform | SIMD Instruction Set | Vector Width | Speed Gain |
|----------|---------------------|--------------|------------|
| **x86_64 (Desktop)** | AVX2 | 8 floats | ~4-6× |
| **ARM64 (Mobile)** | NEON | 4 floats | ~2-3× |
| **WASM** | WASM SIMD | 4 floats | ~2-3× |

**Critical Operations to Vectorize:**

1. **FFT (Butterfly Operations):**
   - KissFFT is already optimized, but custom SIMD FFT could help
   - **Effort:** High (8-10 days), **Gain:** ~30% on FFT
   - **Decision:** Defer to v1.3, KissFFT is sufficient

2. **IIR/FIR Filter Application:**
   - **IIR:** Limited SIMD benefit (sequential dependency)
   - **FIR:** Excellent SIMD candidate (convolution = dot product)
   - **Effort:** Medium (4-5 days), **Gain:** ~3-4× on FIR

3. **Vector Operations (multiply, add, copy):**
   - Used in windowing, scaling, mixing
   - **Effort:** Low (2-3 days), **Gain:** ~4-8× on vector ops

**Implementation Strategy:**

1. **Phase 1 (v1.2.0):** Core vector operations + FIR filtering
   ```cpp
   // native/src/core/simd.hpp
   namespace simd {
       void multiply(float* dst, const float* src1, const float* src2, size_t len);
       void add(float* dst, const float* src1, const float* src2, size_t len);
       float dot_product(const float* a, const float* b, size_t len);
   }
   ```

2. **Phase 2 (v1.3.0):** Custom FFT with SIMD

**Code Example (AVX2):**
```cpp
#include <immintrin.h>

void multiply_avx2(float* dst, const float* src1, const float* src2, size_t len) {
    size_t i = 0;
    for (; i + 8 <= len; i += 8) {
        __m256 a = _mm256_loadu_ps(src1 + i);
        __m256 b = _mm256_loadu_ps(src2 + i);
        __m256 c = _mm256_mul_ps(a, b);
        _mm256_storeu_ps(dst + i, c);
    }
    // Scalar fallback for remainder
    for (; i < len; ++i) {
        dst[i] = src1[i] * src2[i];
    }
}
```

**Platform Detection:**
```cpp
// Runtime CPU feature detection
if (cpu_supports_avx2()) {
    use_avx2_path();
} else if (cpu_supports_sse2()) {
    use_sse2_path();
} else {
    use_scalar_path();
}
```

**Implementation Effort:** 5-7 days (foundation for v1.2.0)

### 6. Unity Burst Compatibility

**Overview:**
Unity Burst Compiler translates C# to highly optimized native code using LLVM. Enables C#-only performance gains without native plugins.

**Challenges:**
- Burst requires pure C# (no P/Invoke)
- SpectraCore is native C++ library
- **Incompatible** with current architecture

**Potential Approaches:**

1. **Hybrid Approach (Recommended for v1.2.0):**
   - Keep native C++ library for complex operations (filter design)
   - Add Burst-compatible C# implementations for hot paths:
     - Filter application (IIR/FIR sample-by-sample)
     - Vector operations
     - Windowing functions

   ```csharp
   [BurstCompile]
   public struct BurstIIRFilter : IJob {
       [ReadOnly] public NativeArray<float> B;
       [ReadOnly] public NativeArray<float> A;
       [ReadOnly] public NativeArray<float> Input;
       public NativeArray<float> Output;

       public void Execute() {
           // Burst-optimized IIR filter application
           // (coefficients pre-computed by native library)
       }
   }
   ```

2. **Full Burst Port (v2.0 consideration):**
   - Rewrite entire library in Burst-compatible C#
   - **Effort:** 100+ days (entire rewrite)
   - **Benefit:** Easier distribution, IL2CPP compatible
   - **Drawback:** Lose C++ ecosystem, harder to maintain

**v1.2.0 Recommendation:**
- Add Burst-compatible filter application (C# implementation)
- Keep native C++ for filter design
- **Best of both worlds:** Design in native, apply in Burst

**Implementation Effort:** 6-8 days (hybrid approach for key operations)
