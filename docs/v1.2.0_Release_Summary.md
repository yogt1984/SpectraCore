## 🎯 Release Highlights

### 1. Bessel (Thomson) Filters ⭐ NEW
**The missing filter type for optimal transient response**

- **Maximally flat group delay** - Linear phase response in passband
- **Minimal overshoot** (<1% in step response vs 5-10% for Butterworth)
- **Excellent transient preservation** - Ideal for pulse signals, drums, speech
- **All filter types supported**: Lowpass, Highpass, Bandpass, Bandstop
- **Orders 1-10** with pre-computed Bessel polynomial poles
- **MATLAB-compatible API**: `DSP.Bessel(order, freq, type)`

**Use Cases:**
- Pulse and transient preservation (drums, percussion, speech plosives)
- Data transmission (minimal intersymbol interference)
- Scientific instrumentation (waveform fidelity)
- Audio applications requiring phase linearity
- Radar/sonar signal processing

**Key Characteristics:**
| Characteristic | Butterworth | Bessel |
|----------------|-------------|--------|
| Passband | Maximally flat | Smooth |
| Group delay | Moderate variation | **Flattest** (linear) |
| Step overshoot | ~5-10% | **<1%** |
| Rolloff | Moderate | Slower |
| Best for | Frequency separation | **Time-domain fidelity** |

## 🔧 Technical Improvements

### Bug Fixes
1. **Bilinear Transform Correction**
   - Fixed critical bug in second-order section bilinear transform
   - Corrected K-factor formula: `K = a2 + 2*a1 + 4` (was incorrect)
   - Fixed a2 coefficient calculation
   - Result: Stable filters across all frequency ranges

2. **Gain Normalization**
   - New helper functions: `compute_dc_gain()` and `compute_nyquist_gain()`
   - Direct evaluation of H(z) at z=1 (DC) and z=-1 (Nyquist)
   - Applies to ALL IIR filters (not just Bessel)
   - Result: Unity gain at DC for lowpass, unity gain at Nyquist for highpass

### Performance
- **Bessel filters**: O(order²) complexity, similar to Butterworth
- **Mobile**: Real-time capable on all modern devices (2018+)
  - FFT: < 5% CPU load for 2048 samples @ 44.1kHz
  - Filters: < 10% CPU load
- **WebGL**: Comparable to native with -O3 optimization

## 🚀 Migration Guide (v1.1 → v1.2)

### No Breaking Changes ✅
**v1.2.0 is 100% backward compatible with v1.0 and v1.1**

### New Features You Can Use

#### 1. Bessel Filters
```csharp
// NEW: Bessel lowpass for minimal overshoot
var (b, a) = DSP.Bessel(4, 0.3f, FilterType.Lowpass);

// NEW: Bessel bandpass for transient-rich signals
var (b2, a2) = DSP.Bessel(3, 0.2f, 0.5f, FilterType.Bandpass);

// Process with minimal phase distortion
float[] output = DSP.FiltFilt(b, a, signal);
```

#### 2. Mobile Deployment
```
1. Build Settings → Select iOS or Android
2. Build → Deploy to device
3. Works exactly like desktop version
```

#### 3. WebGL Deployment
```csharp
// Same code works in browser!
#if UNITY_WEBGL && !UNITY_EDITOR
    // Automatically uses WebAssembly version
#endif

var (b, a) = DSP.Butter(4, 0.3f, FilterType.Lowpass);
// Works in WebGL without changes
```

### Performance Considerations

**Bessel vs Butterworth:**
- Use **Bessel** when: Transient fidelity matters (drums, pulses, speech)
- Use **Butterworth** when: Sharp frequency separation needed
- Bessel has ~10% slower rolloff but much better step response

**Mobile Optimization:**
- Prefer filter orders 4-6 for real-time processing
- Use StreamingFilter for low-latency audio
- Reuse filter objects (don't recreate in audio callback)

## 🐛 Known Issues

### iOS
- Native library must be built on macOS (requires Xcode)
- Build infrastructure complete, but library not pre-built
- **Workaround**: Follow `docs/CrossPlatformBuildGuide.md` on macOS

### High-Frequency Filters
- Orders > 6 at frequencies > 0.7 may have numerical precision issues
- **Workaround**: Use lower orders or lower frequencies
- Affects all IIR filter types, not specific to Bessel

### WebGL
- Performance ~20-30% slower than native (inherent to WebAssembly)
- **Mitigation**: Use -O3 optimization (already enabled)
